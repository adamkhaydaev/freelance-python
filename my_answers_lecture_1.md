## Упражнение 1. Классификация заказов

Типы: 1 — Скрипты автоматизации · 2 — Парсинг/сбор данных · 3 — Отчёты и аналитика · 4 — Боты (Telegram/Discord) · 5 — Интеграции

**№1:** тип = 1 (Скрипты автоматизации); основной потому что задача — автоматически выполнить цепочку действий (зайти в кабинет, скачать, разложить файлы), это не сбор данных ради данных, а автоматизация рутины; не хватает: логин/пароль и способ авторизации на сайте налоговой (там вход по ЭЦП/Госуслугам — это сильно меняет сложность).

**№2:** тип = 2 (Парсинг/сбор данных); основной потому что суть — собрать объявления с сайтов в таблицу, это классический парсинг; не хватает: какие именно два сайта (адреса) и нужна ли разовая выгрузка или регулярная.

**№3:** тип = 4 (Боты Telegram/Discord); основной потому что точка входа и взаимодействие с пользователем — телеграм-бот; запись в Google Sheet здесь второстепенна (это часть работы бота). На стыке с типом 5 (интеграция с Google Sheets), но бот — основное; не хватает: как из фото чека брать данные — вручную пользователь вводит или нужно распознавание (OCR)?

**№4:** тип = 3 (Отчёты и аналитика); основной потому что речь о дашборде с метриками; не хватает: ОПИСАНИЕ НЕПОЛНОЕ — не указано, какие метрики, откуда брать данные, где должен жить дашборд. «Дизайн на ваш вкус» = красный флаг, нет конкретики.

**№5:** тип = 5 (Интеграции); основной потому что задача — соединить две готовые системы (Bitrix24 → Mailchimp), чтобы данные перетекали автоматически; не хватает: есть ли доступы (API-ключи) к обеим системам и по каким полям определять «нужный список».

**№6:** тип = 2 (Парсинг/сбор данных); основной потому что суть — извлечь данные из 800 PDF и свести в Excel (сбор/извлечение данных), хоть источник не сайт, а файлы; не хватает: договоры одного шаблона или разные форматы (от этого зависит, сработает ли автоизвлечение или нужен разбор каждого).

**№7:** тип = 1 (Скрипты автоматизации); основной потому что главное — автоматическая ночная цепочка по расписанию (забрать из 1С → сделать отчёт → отправить почтой). На стыке с типом 3 (отчёт), но автоматизация и расписание — суть задачи; не хватает: как подключаться к 1С (выгрузка/база/API) и доступ к почте для отправки.

**№8:** тип = 4 (Боты Telegram/Discord); основной потому что просят «бота», но что он должен делать — непонятно; не хватает: ОПИСАНИЕ НЕПОЛНОЕ — «умеет всё, что нужно клиентам» = красный флаг, нет ни одной конкретной функции. Без ТЗ начинать нельзя.

**№9:** тип = 2 (Парсинг/сбор данных); основной потому что суть — парсить Avito и складывать в таблицу регулярно (раз в неделю); не хватает: что значит «мой сегмент» — какие категории/фильтры/ключевые слова отслеживать.

**№10:** тип = 3 (Отчёты и аналитика); основной потому что задача — разобрать файл с ошибками и понять причину (анализ данных), а не написать скрипт или собрать данные. Разовая аналитическая задача; не хватает: в каком формате файл ошибок и есть ли доступ к самой CRM для проверки, что именно не загрузилось.

## Упражнение 2. Карта пайплайна

```
Заказ A — парсинг товаров
-------------------------
Вход:
  500 карточек товаров с сайта "Магазин-X" (HTML-страницы каталога/товаров).
  Источник — сам сайт, доступ по ссылкам. Формат на входе — веб-страницы.
  Запуск ручной (раз в неделю по команде), расписание не нужно.

Обработка:
  1. Собрать список ссылок на все 500 карточек (обойти каталог / пагинацию).
  2. Зайти на каждую карточку, вытащить поля: артикул, название, цена,
     наличие, ссылка, ссылка на главное фото.
  3. Записать пропуски: если поле не найдено — пометить ячейку и занести
     товар в лог ошибок (не молча пропускать).
  4. Сложить все строки в таблицу, проверить на дубли по артикулу.
  5. Сохранить результат в Excel + отдельный файл со списком проблемных карточек.

Выход:
  tovary_magazin_x.xlsx — таблица: артикул, название, цена, наличие,
    ссылка, фото (по строке на товар).
  errors_magazin_x.txt — список карточек, где что-то не спарсилось
    (ссылка + какое поле пропущено).

Готово, если:
  1. В tovary_magazin_x.xlsx ровно столько строк, сколько товаров на сайте
     (≈500), и в шапке все 6 полей.
  2. Открыл 3–5 случайных строк, прошёл по ссылке — данные совпадают с сайтом.
  3. Файл errors_magazin_x.txt существует: либо пустой (всё спарсилось),
     либо в нём перечислены конкретные проблемные карточки.
```

```
Заказ B — отчёт по менеджерам
-----------------------------
Вход:
  Выгрузка CRM в формате CSV, ~50 000 строк.
  Источник — файл от заказчика (одна выгрузка). В строках: менеджер,
  дата, тип события (звонок/сделка), сумма сделки.

Обработка:
  1. Прочитать CSV, проверить кодировку и разделитель (; или ,).
  2. Разбить данные по неделям (последние 8 недель) и по менеджерам.
  3. Для каждого менеджера в каждой неделе посчитать: число звонков,
     выручку (сумма закрытых сделок), конверсию = закрытые сделки / звонки.
  4. Строки с битыми/пустыми данными (нет даты, нет менеджера) вынести
     в лог пропусков, чтобы не искажали цифры.
  5. Собрать Excel: отдельный лист или раздел на каждую неделю.

Выход:
  otchet_menedzhery.xlsx — 8 разделов (по неделям), в каждом таблица:
    менеджер, звонки, выручка, конверсия.
  errors_otchet.txt — строки, которые не попали в расчёт (и почему).

Готово, если:
  1. В отчёте ровно 8 недельных разделов, в каждом — все менеджеры.
  2. Пересчитал конверсию руками для 1 менеджера за 1 неделю — сходится.
  3. Сумма звонков+сделок из отчёта + строки из errors_otchet.txt
     = общее число строк в исходном CSV (ничего не потерялось).
```

```
Заказ C — Telegram-бот для сотрудников
--------------------------------------
Вход:
  Команда от сотрудника в Telegram: /отчёт ДД.ММ
  Данные о продажах — из Google Sheets (обновляется вручную).
  Бот обслуживает 12 сотрудников (привязка по Telegram-аккаунту/имени).

Обработка:
  1. Принять команду, проверить формат даты ДД.ММ.
  2. Если формат неверный — ответить подсказкой ("Формат: /отчёт 25.06").
  3. Определить, кто написал (какой из 12 сотрудников).
  4. Найти в Google Sheets продажи этого сотрудника за указанную дату.
  5. Если данные есть — вернуть сумму/список продаж;
     если за эту дату данных нет — ответить "За 25.06 данных нет"
     (а не пустоту и не ошибку).

Выход:
  Сообщение в чат сотруднику с его продажами за день (текстом).
  (Файлы не создаются — выход бота это ответ в Telegram.)

Готово, если:
  1. /отчёт с верной датой и существующими данными возвращает корректную
     сумму (сверил с Google Sheets вручную).
  2. /отчёт с неверным форматом (например "/отчёт 2026") даёт понятную
     подсказку, а не падает.
  3. /отчёт на дату без данных отвечает "данных нет", а не ошибкой.
```

```
Заказ D — выгрузка заказов Wildberries
--------------------------------------
Вход:

  Источник — API (нужен ключ продавца). Запуск автоматический,

Обработка:
  1. По расписанию (ночью) обратиться к API WB, запросить заказы за сутки.

Выход:

Готово, если:
  1. Файл за вчера появился в Dropbox с датой в имени.
  2. В день без заказов файл всё равно создан и содержит строку заголовков.
  3. В log_wb.txt есть запись о последнем запуске (успех или причина сбоя).
```

## Упражнение 3. Красные флаги и уточняющие вопросы

**Сообщение 1:** «Сделай бот, как у Тинькофф, только лучше. Бюджет небольшой.»
- Красный флаг: «как у Тинькофф, только лучше» при небольшом бюджете — несопоставимость хотелок и денег (банк делал бота годами командой).
- Вопрос: Какие 3–5 конкретных функций бот должен уметь в первой версии? (ответ превращает «как у Тинькофф» в реальный список задач и сразу меняет объём)

**Сообщение 2:** «Нужен парсер. Полей не помню, сделай как обычно.»
- Красный флаг: «полей не помню» — у заказчика нет ТЗ, не определены данные для сбора.
- Вопрос: С какого сайта парсим и какие именно поля нужны на выходе (перечислите)? (без этого вообще нельзя оценить — 3 поля с одной страницы или 20 полей с авторизацией это разный объём)

**Сообщение 3:** «Срок завтра, оплата после результата. Доступы дам в процессе.»
- Красный флаг: «оплата после результата» + срочность + нет доступов — риск работать бесплатно и не успеть из-за чужих задержек.
- Вопрос: Какие именно доступы нужны и когда вы сможете их предоставить — сегодня или в процессе? (если доступы придут поздно, «срок завтра» физически невыполним, и это меняет всё)

**Сообщение 4:** «Просто красивый отчёт. Я не разработчик, мне сложно объяснить.»
- Красный флаг: «просто красивый отчёт» — нет конкретики, «красивый» у всех своё, не сказано что в отчёте.
- Вопрос: Какие данные должны быть в отчёте и откуда их брать (какой файл/система)? (отчёт по 3 цифрам из готовой таблицы и аналитика из сырой выгрузки на 50к строк — несопоставимый объём)

**Сообщение 5:** «Сделай интеграцию двух систем. Документации нет, разберёшься по ходу.»
- Красный флаг: «документации нет, разберёшься по ходу» — без API-документации интеграция может оказаться вообще невозможной.
- Вопрос: Какие две системы соединяем и есть ли у них API (открытый доступ к данным)? (если у системы закрытый API или его нет — задача из «средней» превращается в «невыполнимую», это критично для оценки)

## Упражнение 4. Definition of Done

**Заказ:** Telegram-бот для продавца на FunPay, который присылает уведомление о каждом новом заказе и пишет статистику продаж за день по команде.

```
DoD
1) Цель/польза:
   Продавец не сидит в браузере постоянно — бот мгновенно сообщает о новом
   заказе в Telegram, чтобы быстрее выдать товар и не терять покупателей.

2) Вход:
   - Доступ к аккаунту FunPay продавца (cookie/golden_key для чтения заказов).
   - Telegram-аккаунт продавца (chat_id, куда слать уведомления).
   - Команда /стата от продавца для запроса статистики за сегодня.

3) Выход:
   - funpay_bot.py — исходный код бота.
   - config.txt — файл с настройками (ключ FunPay, токен бота, chat_id).
   - README.txt — как запустить и где вписать свои ключи.
   - sales_log.csv — лог продаж (дата, время, товар, сумма) для статистики.

4) Правила:
   1. Каждые N секунд бот проверяет FunPay на новые заказы.
   2. При новом заказе — шлёт в Telegram: товар, сумму, имя покупателя.
   3. Каждую продажу дописывает строкой в sales_log.csv.
   4. По команде /стата считает из лога продажи за сегодня (число + сумма).
   5. Если FunPay не ответил — пробует снова, в чат ошибками не спамит.

5) Критерии качества:
   - Полнота: ни один новый заказ не пропущен (каждый = одно уведомление).
   - Точность: сумма в /стата сходится с реальными продажами за день.
   - Без дублей: один заказ не присылается дважды.
   - Валидность: /стата на день без продаж отвечает "0 продаж", не падает.

6) Ограничения:
   - Зависит от доступа к FunPay (если ключ протух — бот не видит заказы).
   - Зависит от того, не блокирует ли FunPay частые запросы (лимиты).
   - Бот работает, только пока запущен на ПК/хостинге продавца.

7) Не входит:
   - НЕ автоматическая выдача товара покупателю (только уведомление).
   - НЕ автоответы покупателям в чат FunPay.
   - НЕ статистика за неделю/месяц (только за текущий день).
   - НЕ размещение бота на хостинге (отдаётся код + инструкция запуска).
```

## Упражнение 5. Минимум сдачи

Комплект файлов для типичного заказа «парсинг + Excel-выгрузка»:

```
1) tovary.xlsx          — главный результат: таблица со всеми спарсенными
                          товарами, ради неё и делался заказ.
2) errors.txt           — список карточек, которые не спарсились, и почему;
                          показывает заказчику, что пропущено, а не молча скрыто.
3) README.md            — инструкция: как запустить парсер, куда вписать свои
                          данные, где появится результат; чтобы заказчик
                          смог запустить сам без меня.
4) requirements.txt     — список библиотек для установки; без него код не
                          запустится на чужом компьютере.
5) run.bat              — файл запуска в один клик (для Windows), чтобы
                          заказчик не вводил команды в консоли вручную.
6) config.txt           — настройки в одном месте (ссылка на сайт, число
                          товаров, имя выходного файла); менять параметры,
                          не лазая в код.
7) parser.py            — сам скрипт парсинга; передаётся, чтобы заказчик
                          мог запускать повторно (заказ — раз в неделю).
8) log.txt              — лог последнего запуска (когда запускался, сколько
                          собрано, сколько ошибок); для проверки, что всё ОК.
```
